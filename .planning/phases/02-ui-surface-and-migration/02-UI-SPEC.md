### Copywriting Notes

- The dashboard access alert is the only net-new copy in this phase. The current text reads "Event must be COMMITTED or OCCURRED" — this must be updated to include AWARENESS and use title-case status names for user-friendliness. Source: CONTEXT.md `## Implementation Decisions → Dashboard access gate`.
- Status checkbox labels use `{status.toLowerCase()}` with CSS `capitalize` — "awareness" renders as "Awareness" automatically via the existing pattern.
- No new empty states, error states, or confirmation dialogs are introduced.

## Interaction Contracts

### Filter Checkboxes — `app/events/page.tsx`

- Initial state: all 5 statuses checked (AWARENESS included via `STATUS_DISPLAY_ORDER`)
- Display order: AWARENESS, PIPELINE, COMMITTED, CANCELED, OCCURRED (from `STATUS_DISPLAY_ORDER`)
- Check/uncheck: toggles `selectedStatuses` array; AWARENESS behaves identically to other statuses
- "Clear Filters" button: resets to all 5 statuses checked (replaces current 4-item array with `[...STATUS_DISPLAY_ORDER]`)
- Visual: checkboxes use existing `w-4 h-4 text-blue-600 border-neutral-300 rounded focus:ring-blue-500` class — no change

### Dashboard Access Gate — `app/events/page.tsx`

- Trigger: user clicks "View Dashboard" on an event card
- Allowed statuses (non-manager): COMMITTED, OCCURRED, AWARENESS
- Blocked statuses (non-manager): PIPELINE, CANCELED
- On allowed: `router.push(/events/${event.slug || event.id}/dashboard)` — no change
- On blocked: `alert(...)` with updated copy (see Copywriting Contract above)
- Modal button render condition: same logic as click handler — both locations must match

### Status Dropdown — `app/events/[id]/settings/page.tsx`

- AWARENESS added as first `<option>` before Pipeline
- Option order: Awareness, Pipeline, Committed, Occurred, Canceled
- No other options changed
- Claude's discretion (from CONTEXT.md): whether to refactor remaining options to use `STATUS_DISPLAY_ORDER` dynamically — recommended yes for consistency, but not required by any requirement

### Database Migration

- No UI interaction contract — pure backend operation
- `npx prisma migrate dev --name add-awareness-status` produces a no-DDL migration file
- No visual output for the user
